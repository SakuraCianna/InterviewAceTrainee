# 面霸练习生需求文档

## 0. 文档信息

- 文档状态：需求草案
- 当前版本：v0.3
- 适用阶段：首版 MVP 设计、技术选型、后续开发计划拆分
- 主要用途：明确产品边界、核心流程、技术约束、后台管理、日志追溯和上线验收标准

## 1. 项目概述

面霸练习生是一个多场景 AI 语音模拟面试平台。首版产品覆盖工作面试、研究生复试面试、考公面试和雅思口语面试，通过轻量 2D 数字人面试官、语音问答、AI 追问和完整面试报告，帮助用户提前适应不同类型面试的表达节奏和追问压力。

首版目标不是做真人级数字人系统，也不是只做单一技术面试工具，而是先做一个可小范围收费验证的多场景 AI 面试训练 MVP。

## 2. 产品定位

### 2.1 目标用户

- 准备实习、秋招、春招或社招的求职者
- 准备研究生复试的学生
- 准备公务员或事业单位结构化面试的考生
- 准备雅思口语考试的用户
- 希望用语音方式模拟真实面试的人

### 2.2 核心卖点

- 全程语音模拟真实面试
- 自研轻量数字人面试官，降低数字人成本
- 支持工作面试、研究生复试、考公面试、雅思口语四个模块
- 不同模块支持不同轮次和不同收费标准
- 面试结束后生成完整复盘报告
- 按面试次数收费，前期由管理员人工开通次数
- AI 服务首版固定为 DeepSeek LLM、腾讯云 ASR、腾讯云 TTS，后台仅展示状态、启停和调用日志，避免运营误改供应商

## 3. 首版 MVP 范围

### 3.1 首版包含

- 用户邮箱注册和登录
- 普通用户支持邮箱验证码登录和密码登录
- 管理员支持密码 + 邮箱验证码双重认证
- 管理员账号同时拥有普通用户能力和管理员能力
- 管理员账号面试次数不受限制
- 用户面试次数余额管理
- 管理员手动给用户增加、扣减、修正、退回面试次数
- 轮流语音面试
- 自研轻量 2D 数字人面试官
- 工作面试模块
- 研究生复试面试模块
- 考公面试模块
- 雅思口语面试模块
- 工作面试开始前上传简历，填写应聘岗位和岗位要求
- 研究生复试开始前填写报考专业，可补充研究方向
- 不同模块的轮次、评分维度和消耗次数配置
- 面试结束后保存并展示完整报告
- LLM 固定使用 DeepSeek V4 Flash
- ASR/TTS 固定使用腾讯云语音服务
- 完整日志和审计能力
- Docker Compose + Nginx 部署

### 3.2 首版不包含

- 在线支付
- 手机验证码
- App 或小程序
- 真人级数字人视频
- 实时可打断语音通话
- 多人面试
- 企业招聘端
- 自动发票、自动退款

### 3.3 MVP 优先级

首版需求按优先级分为 P0、P1、P2。

P0 是必须完成的最小可收费闭环：

- 邮箱注册和登录
- 普通用户次数余额
- 管理员后台登录
- 管理员手动发放和修正次数
- 轮流语音面试
- 基础轻量数字人
- DeepSeek V4 Flash 接入
- 腾讯云 ASR 和腾讯云 TTS 接入
- 面试报告生成和保存
- 工作面试资料预分析
- 研究生复试专业背景设置
- 次数账本、面试会话日志、AI 调用日志、管理员审计日志

P1 是增强体验和运营效率的能力：

- 固定 AI 服务状态检查和失败日志记录
- 后台测试供应商连通性
- 更细的报告维度评分
- 用户历史面试记录筛选
- 后台用户搜索和日志筛选
- Nginx Basic Auth 或 IP 限制

P2 是首版上线后再考虑的扩展：

- 在线支付
- 自动退款
- 高级套餐
- Live2D 或更复杂数字人
- 静音自动检测
- 实时可打断语音对话
- 图片简历 OCR 和项目经历优化

### 3.4 首版验收标准

首版 MVP 需要满足以下验收标准：

- 用户可以通过邮箱验证码注册并登录
- 用户可以设置密码，并通过邮箱 + 密码登录
- 管理员必须通过密码 + 邮箱验证码进入后台
- 管理员可以给普通用户增加、扣减、修正和退回面试次数
- 普通用户开始有效面试时，根据所选模块扣除对应次数
- 管理员发起面试不扣次数
- 工作面试必须先上传简历并填写目标岗位、岗位要求
- 研究生复试必须先填写报考专业，研究方向可选填
- 面试过程必须支持 AI 语音提问、用户语音回答、用户手动点击回答完毕
- 面试结束后必须生成完整报告
- 用户可以查看自己的历史报告
- 管理员可以查看所有用户的面试记录、报告和关键日志
- LLM、ASR、TTS 至少各有一种可用配置
- AI 调用失败时必须记录失败原因
- 所有次数变更必须能从账本中追溯
- 所有管理员敏感操作必须能从审计日志中追溯

## 4. 用户角色

### 4.1 普通用户

普通用户可以注册、登录、查看剩余面试次数、进入语音面试、查看历史面试记录和完整面试报告。

普通用户每开始一次有效面试，根据所选模块消耗对应面试次数。

### 4.2 管理员

管理员是拥有 `admin` 角色的用户账号。管理员账号不是单独的一套登录体系，而是在同一个账号系统内拥有更高权限。

管理员既可以进入普通用户前台，也可以进入管理员后台。同一个管理员账号在普通前台使用面试功能时，不受面试次数限制。

管理员能力包括：

- 登录普通用户前台
- 使用普通用户的全部功能
- 不受面试次数限制
- 进入管理员后台
- 管理用户
- 管理用户面试次数
- 查看面试记录
- 查看完整日志
- 查看并启停固定 AI 服务配置
- 调整系统配置

管理员虽然不受次数限制，但管理员的面试行为、后台操作和配置修改仍然必须记录日志，确保可追溯。

## 5. 登录与认证需求

### 5.1 普通用户登录

普通用户支持两种登录方式：

- 邮箱验证码登录
- 邮箱 + 密码登录

用户可以只使用邮箱验证码登录，也可以设置密码后使用密码登录。

密码字段允许为空。若用户未设置密码，则只能通过邮箱验证码登录。

### 5.2 管理员登录

管理员后台必须使用双重认证：

1. 输入管理员邮箱和密码
2. 密码校验通过后，系统发送邮箱验证码
3. 验证码校验通过后，签发管理员登录 token

管理员登录要求：

- 登录邮箱必须属于管理员账号
- 密码必须正确
- 邮箱验证码必须正确且未过期
- 后台 token 有效期应短于普通用户 token
- 所有后台接口必须再次校验 `admin` 权限

### 5.3 密码存储

密码不得明文保存，也不得使用可逆加密保存。

密码必须使用单向哈希保存，推荐使用：

- Argon2id
- Python 库优先使用 `pwdlib`

性能要求：

- 单次密码 hash 或 verify 建议控制在 100ms 到 300ms 左右
- 登录接口必须限流
- 密码校验应避免阻塞 FastAPI 主事件循环

## 6. 面试流程

### 6.1 面试模式

首版采用轮流说话模式：

1. AI 面试官语音提问
2. 用户语音回答
3. 用户点击“回答完毕”
4. 系统进行语音识别
5. LLM 生成追问、评价或下一题
6. TTS 合成 AI 面试官语音
7. 数字人播放语音并做口型和状态变化
8. 重复以上流程直到面试结束

首版不做自动打断，不做真人视频通话式实时对话。

### 6.2 面试模块

首版支持四种面试模块：

- 工作面试
- 研究生复试面试
- 考公面试
- 雅思口语面试

### 6.3 模块轮次与收费规则

不同模块采用不同的轮次和次数消耗规则：

- 工作面试：完整体验包含专业一面、专业二面、HR 面，消耗 3 次
- 研究生复试面试：一趟复试模拟，消耗 1 次
- 考公面试：一场结构化面试模拟，消耗 1 次
- 雅思口语面试：Part 1、Part 2、Part 3 全流程，消耗 2 次

模块规则需要可配置，后续管理员可以在后台调整轮次名称、消耗次数、报告维度和是否启用。

### 6.4 模块内容范围

工作面试覆盖：

- 简历文字提取和岗位 JD 预分析
- 目标岗位匹配度追问
- 岗位专业能力
- 项目经历追问
- 技术基础或岗位技能
- 业务理解
- HR 动机面
- 薪资期望和稳定性表达

研究生复试面试覆盖：

- 报考专业和研究方向背景设置
- 中文或英文自我介绍
- 专业基础
- 科研兴趣
- 毕业设计或本科项目
- 读研规划
- 导师沟通问题

考公面试覆盖：

- 综合分析
- 组织协调
- 人际沟通
- 应急应变
- 岗位认知
- 语言表达稳定性

雅思口语面试覆盖：

- Part 1 日常问答
- Part 2 个人陈述
- Part 3 深入讨论
- 流利度
- 词汇
- 语法
- 发音和连贯性

## 7. 轻量数字人需求

### 7.1 数字人定位

首版采用自研轻量 2D 数字人，不使用真人级数字人 API。

数字人风格应接近 Google / Material 风格：

- 干净
- 克制
- 现代
- 职业
- 低压迫感
- 不直接复制 Google 品牌元素

### 7.2 数字人状态

数字人至少支持以下状态：

- 空闲
- 说话中
- 聆听中
- 思考中
- 正向反馈
- 严肃反馈
- 错误或异常状态

### 7.3 嘴型同步

首版采用低成本嘴型同步方案：

- TTS 音频播放时，通过 Web Audio API 获取音量变化
- 根据音量驱动嘴巴开合
- 后续可升级为音素级口型同步或 Live2D

## 8. 面试次数与人工收费流程

### 8.1 商业模式

首版采用按次数收费，但不接在线支付。

运营流程：

1. 用户通过社交平台看到产品
2. 用户注册账号
3. 用户联系项目方购买面试次数
4. 管理员在后台手动给用户增加次数
5. 用户消耗次数进行面试

### 8.2 扣次规则

普通用户开始一次有效面试时，系统扣除 1 次面试次数。

若面试因系统故障失败，管理员可以后台退回次数。

管理员账号不受面试次数限制，不因面试扣除次数。

管理员发起的面试仍然需要创建面试会话和报告，但次数账本中不产生扣次记录。面试会话应标记为管理员无限制使用，方便后续区分真实用户消耗和管理员测试行为。

### 8.3 次数账本

用户剩余次数可以冗余存储在用户表中，但所有次数变动必须写入账本。

账本是处理纠纷、退款、补偿和运营统计的最终依据。

## 9. 面试报告需求

### 9.1 报告生成

每次成功完成的面试都必须生成完整报告。

报告应保存到数据库，并允许用户后续查看。

### 9.2 报告内容

完整报告至少包含：

- 总评分
- 分维度评分
- 技术准确性评价
- 表达清晰度评价
- 追问应对能力评价
- 项目深度评价
- 面试完整度评价
- 本次表现优点
- 本次主要短板
- 每个关键问题的回答点评
- 推荐修改后的回答
- 后续可能被追问的问题
- 下一步练习建议

### 9.3 报告可见性

普通用户可以查看自己的报告。

管理员可以查看所有用户的报告，用于售后、纠纷处理和产品质量分析。

## 10. AI 服务配置

### 10.1 总体原则

LLM、ASR、TTS 首版采用固定供应商方案：

- LLM：DeepSeek V4 Flash
- ASR：腾讯云语音识别
- TTS：腾讯云语音合成

后台不再提供新增任意模型配置入口，只保留固定 AI 服务的状态查看、启停、连通性测试和调用日志。这样可以降低误配置风险，更适合当前小范围商业验证阶段。

### 10.2 LLM

首版默认 LLM：

- DeepSeek V4 Flash

首版优先使用国内模型，不依赖国外模型。

### 10.3 ASR

ASR 表示语音识别服务，用于将用户语音转成文字。

首版固定使用腾讯云 ASR。

### 10.4 TTS

TTS 表示语音合成服务，用于将 AI 面试官文本转成语音。

首版固定使用腾讯云 TTS。

### 10.5 AI 路由能力

系统需要提供统一的 AI 服务层：

- LLMProvider
- ASRProvider
- TTSProvider
- AIServiceRouter

AIServiceRouter 负责：

- 根据服务类型选择固定供应商
- 校验固定配置是否启用
- 失败时记录可追溯错误
- 记录调用日志
- 统计调用耗时
- 统计 token 或音频时长
- 估算调用成本

首版路由策略：

- 只选择 `enabled = true` 的配置
- 优先选择用途匹配的配置
- 优先级数字越小越优先
- 固定供应商失败时记录错误并提示用户稍后重试

## 11. 管理员后台需求

### 11.1 后台形态

前台和后台共用同一个 React 项目。

后台入口路径不使用默认 `/admin`，而是通过环境变量配置一个不易猜测的路径。

隐藏路径只是降低暴露概率，不作为核心安全方案。

核心安全仍然依赖：

- 管理员双重认证
- 后端权限校验
- 接口限流
- 操作日志
- 可选 Nginx Basic Auth 或 IP 限制

### 11.2 用户管理

后台需要支持：

- 查看用户列表
- 搜索用户
- 查看用户详情
- 查看用户剩余次数
- 查看用户面试记录
- 查看用户报告
- 禁用或启用用户

### 11.3 次数管理

后台需要支持：

- 给用户增加次数
- 扣减用户次数
- 修正用户次数
- 给用户退回次数
- 填写操作原因
- 查看次数账本

所有次数操作都必须写入账本和管理员审计日志。

### 11.4 AI 配置管理

后台需要支持：

- 查看固定 AI 服务配置
- 启用配置
- 禁用配置
- 测试连通性
- 查看最近调用成功率和失败原因

API Key、Secret Key 等敏感信息必须加密存储，后台不得明文回显完整密钥。

### 11.5 系统配置管理

后台需要支持配置：

- 是否开放注册
- 新用户默认赠送次数
- 单次面试最大轮数
- 单次面试最长时间
- 单轮用户回答最长时间
- 是否允许普通用户密码登录
- 是否允许验证码登录
- 默认 LLM 配置固定为 DeepSeek V4 Flash
- 默认 ASR 配置固定为腾讯云 ASR
- 默认 TTS 配置固定为腾讯云 TTS

## 12. 日志与可追溯需求

日志必须全面，重点服务于：

- 退款
- 纠纷处理
- 次数异常排查
- AI 调用失败排查
- 管理员误操作追踪
- 成本统计
- 服务质量分析

### 12.1 次数账本日志

记录所有用户次数变化。

建议字段：

- 用户 ID
- 变化数量
- 变化后余额
- 变动原因
- 关联面试 ID
- 操作管理员 ID
- 操作备注
- 创建时间

变动原因包括：

- 管理员发放
- 面试扣次
- 系统故障退款
- 手动修正
- 活动赠送

### 12.2 面试会话日志

记录一次面试的完整生命周期。

建议字段：

- 用户 ID
- 面试类型
- 会话状态
- 是否扣次
- 开始时间
- 结束时间
- 失败原因
- 关联报告 ID
- 创建时间
- 更新时间

会话状态包括：

- created
- running
- completed
- failed
- cancelled

### 12.3 AI 调用日志

记录每一次 LLM、ASR、TTS 调用。

建议字段：

- 面试会话 ID
- 供应商类型
- 供应商名称
- 模型名称
- 调用用途
- 供应商 request ID
- 输入摘要
- 输出摘要
- 输入 token 数
- 输出 token 数
- 音频时长
- 调用耗时
- 是否成功
- 错误码
- 错误信息
- 成本估算
- 创建时间

### 12.4 管理员审计日志

记录所有后台敏感操作。

建议字段：

- 管理员用户 ID
- 操作类型
- 目标类型
- 目标 ID
- 操作前快照
- 操作后快照
- IP 地址
- User-Agent
- 创建时间

敏感操作包括：

- 增加用户次数
- 扣减用户次数
- 退款或退回次数
- 启停固定 AI 服务配置
- 禁用用户
- 修改系统配置

### 12.5 音频与隐私

首版不建议默认长期保存完整用户音频。

推荐保存：

- ASR 转写文本
- AI 提问文本
- 用户回答文本
- AI 追问文本
- 面试报告快照
- AI 调用摘要
- 错误信息和供应商 request ID

如后续需要保存音频，必须在用户协议或隐私说明中明确告知。

## 13. 数据库选型

数据库采用 PostgreSQL。

选择原因：

- 适合复杂业务数据
- 支持 JSONB，方便存储 AI 报告和结构化评分
- 支持全文检索扩展
- 当前已接入 pgvector，用于已审核面试能力卡片的向量召回和长期能力画像扩展
- 与 FastAPI、SQLAlchemy、Alembic 配合成熟

- interview_turns

- ai_call_logs

- system_configs

### 15.1 关键数据一致性要求

- 用户余额字段可以作为快速读取字段，但不能作为唯一账本依据
- 次数变化必须先写入账本，再更新用户余额
- 面试会话、面试轮次、AI 调用日志、报告之间必须有关联 ID
- 报告应保存生成时的快照，避免后续 AI 服务参数变化导致历史报告不可解释
- 固定 AI 服务启停必须写入管理员审计日志
- 敏感密钥只保存加密后的密文，不在普通查询接口中返回完整值

## 16. 非功能需求

### 16.1 性能

- 普通页面应快速加载
- 登录、验证码、后台查询接口应有限流
- 密码校验不能明显拖慢整体服务
- AI 调用应设置超时时间
- 面试流程中应显示清晰的聆听、处理中、说话中状态

### 16.2 稳定性

- AI 供应商调用失败时应记录日志
- 固定 AI 服务失败时应保留错误原因并提示稍后重试
- 面试异常失败时应保留会话状态和错误原因
- 系统故障导致扣次但未完成面试时，管理员应能退回次数

### 16.3 安全

- 密码使用 Argon2id 哈希
- 管理员必须双重认证
- 后台 API 必须校验 admin 权限
- 敏感配置加密存储
- API Key 不完整回显
- 登录和验证码接口限流
- 后台敏感操作必须审计

### 16.4 合规与隐私

- 明确告知用户面试内容会被用于生成报告
- 谨慎保存音频
- 默认不长期保存完整音频
- 用户报告仅用户本人和管理员可见
- 后续上线前需要补充用户协议和隐私政策

## 17. 上线验证目标

首版上线后主要验证：

- 是否有人愿意注册试用
- 用户是否愿意联系项目方购买次数
- 语音面试体验是否足够自然
- AI 追问是否有面试感
- 报告是否让用户觉得有价值
- 人工开通次数流程是否可运营
- 供应商成本是否可接受
- 日志是否足够支撑售后和纠纷处理

### 17.1 小范围运营流程

首版推荐采用小范围人工运营：

1. 在社交平台发布产品介绍和试用入口
2. 用户注册账号
3. 用户联系项目方购买或领取面试次数
4. 管理员后台为用户增加次数，并填写备注
5. 用户完成语音面试
6. 用户查看完整报告
7. 若出现异常，管理员根据会话日志、AI 调用日志和次数账本判断是否退回次数

这个流程的目标是先验证真实付费意愿和产品体验，不在首版投入在线支付、自动退款等复杂能力。

## 19. 当前已确认决策

- 前端使用 React
- 后端使用 FastAPI
- Python 使用 uv 管理环境
- Python 版本使用 3.12
- Node.js 使用 24 LTS
- 数据库使用 PostgreSQL
- 面试能力卡片使用 pgvector 做内部向量召回，确定性题库流程仍然负责轮次和兜底
- 部署使用 Docker Compose + Nginx
- 登录使用邮箱验证码和密码体系
- 手机验证码暂不使用
- 在线支付首版不做
- 次数由管理员人工开通
- 管理员账号同时拥有普通用户能力和管理员能力
- 管理员面试次数不受限制
- LLM 默认使用 DeepSeek V4 Flash
- ASR/TTS 默认使用腾讯云语音服务
- AI 服务优先使用国内供应商，首版不开放新增模型配置
- 面试采用轮流说话模式
- 数字人采用自研轻量 2D 方案
- 面试结束后保存并展示完整报告
- 日志必须全面，支持退款、纠纷和审计追溯
