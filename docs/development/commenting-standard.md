# 代码注释规范

本项目参考阿里巴巴 Java 开发规范等工程实践，采用“职责、契约、风险”导向的注释方式。
注释用于解释代码无法自行表达的业务边界和设计原因，不追求逐行注释数量。

## 必须注释

- Java 公开类、接口、枚举、记录：说明职责、输入输出边界或生命周期。
- 对外公开方法：说明调用契约；参数、返回值或异常存在歧义时使用 `@param`、`@return`、`@throws`。
- HTTP 请求/响应 DTO：说明它代表的接口语义，不为每个名称已经自解释的字段重复写注释。
- 事务、行锁、幂等键、Outbox、消息 ACK/重试/DLQ：说明执行顺序以及顺序错误的后果。
- 安全控制：说明信任边界、失败策略、敏感数据是否持久化。
- 资源上限、超时和降级：说明它保护的资源或用户体验。
- 临时兼容方案与迁移点：说明保留原因和移除条件。

## 不应添加

- 对赋值、循环、条件、组件名称的逐字翻译。
- 与实现不一致的历史说明、未经验证的性能结论。
- 密钥、Token、Cookie、生产地址中的凭据或用户隐私数据。
- 用注释隐藏失效代码；无用代码应安全删除并由 Git 保留历史。
- 在 Javadoc 中使用 `<p>`、`<br>`、列表或表格等 HTML 标签；段落直接使用空注释行分隔。
- `@author`、`@date` 等多人协作归属信息；本项目由单人维护，变更历史以 Git 为准。

## 语言与维护

- 业务和设计说明使用简洁中文；协议字段、稳定错误码和标准术语保留英文。
- 修改受注释约束的逻辑时必须同步修改注释和相关架构/运维文档。
- Reviewer 应检查注释是否解释“为什么”，并把误导性注释视为代码缺陷。
